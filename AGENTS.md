# AGENTS.md — Antigravity 에이전트 행동 규칙 (DB 관리 실습용)

> 이 문서는 Antigravity, Claude, Gemini 등 모든 AI 코딩 에이전트가 본 워크스페이스에서 작업할 때 **반드시** 준수해야 하는 메타 규칙입니다.
> 사용자는 **코딩 초보 교육자**이며, 본 워크스페이스는 **Supabase 기반 데이터베이스 관리 실습**을 위한 환경입니다.
> 프로젝트 명세(무엇을 만들지)는 `prd/` 폴더의 11개 문서를 참조하세요.
> **이 파일의 규칙은 prd/ 의 어떤 내용보다도 우선합니다.**

## 0. 우선순위 (Conflict Resolution)

규칙이 서로 충돌할 때는 다음 순서로 따릅니다.
1. 사용자가 직접 채팅으로 내린 즉시 지시
2. 본 `AGENTS.md` 의 규칙
3. `prd/` 폴더 내 명세 문서
4. AI 에이전트의 일반 학습 지식

### 0.1 내부 규칙 충돌 메타 룰
본 `AGENTS.md` **내부 규칙들끼리** 충돌할 경우, 다음 순서로 우선합니다.
1. **안전성 (규칙 2)** — 되돌릴 수 없는 작업은 항상 차단·확인 우선
2. **보안·암호화 (규칙 8)** — 민감 데이터 노출 위험이 있으면 차단
3. 그 외 규칙은 본 문서 순서대로 (1, 3, 4, 5, 6, 7, 9, 10)

예: "단계별 계획(규칙 3)"을 위해 Plan 마크다운을 자세히 적다 보면 한 응답이 길어져 "단계별 진행(규칙 6) 최대 3파일"을 어길 수 있습니다. 이 경우 규칙 6은 코드 작업에만 적용되므로 충돌이 아니지만, 진짜 충돌이 발생하면 위 우선순위로 해결합니다.

## 1. 한국어 응답 (Language)

- 모든 생각(Thought), 사용자 응답, 설명, 주석은 **반드시 자연스러운 한국어**로 작성합니다.
- 변수명·함수명·SQL 키워드 등 코드 내 영문 식별자는 그대로 둡니다.
- 주석은 한국어로 작성하되, 표준 라이브러리/외부 라이브러리의 영문 docstring은 유지합니다.

**왜 이 규칙이 필요한가?**
사용자는 한국어로 수업을 진행하는 교육자입니다. 영문 응답은 즉시 번역 비용을 발생시키고 학생에게 혼선을 줍니다.

## 2. 안전성 우선 (Safety First) — 사전 동의 필수

다음 작업은 **반드시** 사용자에게 "이렇게 변경해도 될까요?"라고 묻고 명시적 승인을 받은 후에만 진행합니다.

- 기존의 잘 동작하던 코드/파일에서 **의미가 달라지는 수정**:
  - 함수 시그니처(이름·인자·반환 타입) 변경
  - 파일·폴더·식별자(변수·함수·테이블) 이름 변경
  - 동작 결과가 바뀌는 로직 변경 (단순 포매팅·주석 추가는 제외)
- 파일 또는 폴더 삭제
- 데이터베이스 파괴적 명령: `DROP TABLE`, `DROP SCHEMA`, `TRUNCATE`, `DELETE` (WHERE 절 없는 경우), `ALTER TABLE ... DROP COLUMN`
- 마이그레이션 롤백 (이미 적용된 마이그레이션을 되돌리는 작업)
- 환경변수 파일(`.env`, `.env.local`) 수정 또는 삭제
- `git` 강제 푸시·리셋·브랜치 삭제

> 단순 코드 포매팅, 주석 추가, 한글화 같이 **동작 의미를 바꾸지 않는 수정**은 사전 동의 없이 진행해도 됩니다. 다만 변경 내역은 응답에 명시합니다.

**왜 이 규칙이 필요한가?**
데이터베이스 실습 환경에서 한 번 잘못된 명령은 학생들의 학습 데이터를 통째로 날릴 수 있습니다. 되돌릴 수 없는 작업은 반드시 사람의 눈으로 한 번 더 검토되어야 합니다.

## 3. 단계별 계획 수립 (Plan Before Code)

복잡한 기능을 구현할 때는, **코드를 짜기 전에 반드시 마크다운 형식의 Implementation Plan**을 먼저 사용자에게 제시합니다.

계획에 포함되어야 하는 항목:
- 무엇을 만들지 (What)
- 왜 필요한지 (Why)
- 어떤 파일을 새로 만들거나 수정할지 (Which files)
- 작업 순서 (Order)
- 검증 방법 (How to verify)

사용자의 "좋아요, 진행하세요" 승인을 받은 후 코드를 작성합니다.

**왜 이 규칙이 필요한가?**
초보 학습자는 코드보다 "흐름"을 먼저 이해해야 합니다. Plan은 학습 자료 그 자체입니다.

## 4. 친절하고 충분한 코드 설명 (Educational Comments)

코드를 작성하거나 수정할 때 다음을 **반드시** 함께 제공합니다.

- 각 함수·블록 위에 **한글 주석**으로 역할 설명
- 코드 블록 아래에 자연어로 "이 코드는 무엇을 하나요?" 설명 문단
- 처음 등장하는 기술 용어는 **한 줄 정의**를 괄호로 병기
  - 예: `RLS (Row Level Security, 행 단위 접근 제어)`
- 사용자가 "왜 이렇게 하나요?"라고 물을 만한 부분은 미리 설명

**금지 사항**:
- 한 줄짜리 영문 주석 (`// fetch data`)만 다는 것
- 코드만 던지고 설명을 생략하는 것

**왜 이 규칙이 필요한가?**
사용자는 코딩 초보이며 교육자입니다. 본인이 이해해야 학생을 가르칠 수 있습니다.

## 5. 코드 생략 절대 금지 (Full Copy-Paste Guarantee)

수정된 코드를 출력할 때 **절대** 다음과 같이 생략하지 않습니다.

- `// ... 기존 코드 ...`
- `// (이하 동일)`
- `# ... unchanged ...`
- `...` 으로 끝나는 함수

대신, **수정된 파일 전체**를 그대로 복사해서 덮어쓸 수 있도록 출력합니다.
파일이 매우 긴 경우에도, 변경된 함수 단위의 **전체 함수 코드**는 빠짐없이 제시합니다.

**왜 이 규칙이 필요한가?**
초보자는 생략된 코드가 정확히 어디에 들어가야 하는지 모릅니다. "복사 → 붙여넣기 → 저장"으로 끝나는 흐름이 학습 진도를 보장합니다.

---

## 6. 단계별 진행 (Step-by-Step Execution)

한 번에 여러 파일이나 복잡한 기능을 한꺼번에 수정하지 않습니다.

- **기존 코드를 수정하는 경우**, 한 번의 응답에서 동시에 다루는 파일은 **최대 3개**까지로 제한합니다.
  - 단, 비어있는 신규 명세 문서·설정 파일·`README` 등을 동시에 *생성*하는 작업은 본 제한에서 제외됩니다 (수정이 아니라 생성이므로).
- 하나의 기능이 완벽하게 동작하는지 사용자에게 **확인을 받은 후** 다음 단계로 넘어갑니다.
- 큰 작업은 STEP 1, STEP 2, ... 형태로 명시적으로 분할합니다.
  - **"큰 작업"의 기준**: 예상 작업이 다음 중 하나 이상을 포함하면 큰 작업입니다.
    - 3개 이상의 의사결정 분기 (예: 어떤 라이브러리 선택, 어떤 스키마 형태 선택 등)
    - 5개 이상의 파일 생성·수정
    - DB 스키마 변경을 동반함

**왜 이 규칙이 필요한가?**
한꺼번에 10개 파일이 바뀌면 어디서 문제가 났는지 학생도 교사도 추적할 수 없습니다.

## 7. Supabase 쿼리 관리 규칙 (Migration Discipline)

### 7.1 파일명 규칙
모든 SQL 마이그레이션 파일은 다음 명명 규칙을 **반드시** 따릅니다.

```
<3자리 번호>_<동사>_<대상>.sql
예시:
  001_create_sentiment_logs_table.sql
  002_add_index_on_created_at.sql
  003_enable_rls_on_sentiment_logs.sql
  004_create_policy_select_own_rows.sql
```

- 번호는 3자리 0-padding (001, 002, ..., 099, 100, ...)
- 동사는 영문 소문자 (`create`, `add`, `alter`, `enable`, `drop`, `rename`)
- 대상은 snake_case
- 한 파일에는 **하나의 변경 의도**만 담습니다 (테이블 생성과 인덱스 추가를 한 파일에 섞지 않음)

### 7.2 파일 위치
모든 마이그레이션 SQL은 **반드시** 다음 경로에 모읍니다.
```
supabase/migrations/
```
프로젝트 곳곳에 .sql 파일이 흩어지지 않게 합니다.

### 7.3 롤백 동반
파괴적 변경(컬럼 삭제 등)을 포함하는 마이그레이션은 같은 이름의 `_rollback.sql` 짝꿍 파일을 함께 만듭니다.
```
005_drop_legacy_column.sql
005_drop_legacy_column_rollback.sql
```

**왜 이 규칙이 필요한가?**
실습 환경에서 마이그레이션이 뒤섞이면 어느 순서로 실행해야 할지 학생이 혼동합니다. 번호와 폴더로 강제 정렬합니다.

## 8. 강력한 보안 및 암호화 (Security First)

### 8.1 개인정보·민감 데이터 처리
다음 데이터는 **반드시** DB 저장 전에 암호화하거나 해시 처리합니다.

| 데이터 종류 | 처리 방식 | 권장 방법 |
|---|---|---|
| 비밀번호 | 단방향 해시 | `bcrypt` (cost 12 이상) |
| 이메일·전화번호 | 양방향 암호화 | `pgcrypto` 의 `pgp_sym_encrypt` |
| 학생 식별 정보 | 양방향 암호화 | `pgcrypto` 또는 앱 레이어 AES-256-GCM |
| 분석 입력 텍스트(개인 감정) | 양방향 암호화 | `pgcrypto` |
| 분석 결과 메타데이터 | 평문 가능 | 단, user_id 와의 조인은 RLS로 보호 |

### 8.2 환경변수
- API Key, DB 비밀번호, 암호화 키는 **절대** 코드나 마이그레이션 SQL에 하드코딩하지 않습니다.
- 모든 비밀값은 `.env` 파일에만 두고, `.gitignore` 에 `.env`가 포함되어 있는지 매번 확인합니다.
- `.env.example` 파일을 별도로 두어 키 이름만 공유합니다.

### 8.3 RLS 의무화
`auth.users` 와 연결되는 모든 테이블은 **생성과 동시에 RLS를 활성화**합니다.
RLS 정책 없는 테이블에 데이터를 넣는 행위는 금지입니다.

**왜 이 규칙이 필요한가?**
교육 환경이라도 학생이 입력한 텍스트(감정·생각)는 매우 민감한 개인정보입니다. 실습 중 유출 사고는 학생 신뢰를 영구히 잃습니다.

## 9. Antigravity 워크플로 특화 규칙

### 9.1 작업 시작 전 (Before You Edit)
에이전트가 파일을 새로 만들거나 수정하기 전에 **반드시** 다음 체크리스트를 마음속으로 확인합니다.

- [ ] 이 작업이 사용자의 **이번 요청**에 포함되어 있는가? (요청 범위 밖이면 멈추고 묻기)
- [ ] 이 파일은 기존에 존재하는가? 존재한다면 먼저 읽었는가?
- [ ] 같은 일을 하는 기존 코드/함수가 있는가? 있다면 재사용해야 함
- [ ] 만들 파일의 위치가 본 규칙 7.2 / `prd/` 명세와 일치하는가?

### 9.2 도구 사용 우선순위
- 파일 읽기: 사용 중인 에이전트의 **파일 읽기 도구**(Read / Open / `cat` 등) 우선. 한 파일을 통째로 출력시키는 셸 명령은 마지막 수단.
- 검색: **Grep / Glob 또는 동등 도구** 사용. 전체 파일 덤프 금지 (불필요한 토큰 소모).
- 실행: 사용자에게 명령어를 먼저 보여주고 "실행할까요?" 확인 후 진행.

> 도구 이름은 에이전트 환경에 따라 다를 수 있습니다(Antigravity·Claude Code·Cursor 등). 본 규칙은 **이름이 아니라 기능**을 기준으로 합니다.

### 9.3 자동 실행 금지 명령
다음 명령은 사용자 확인 없이 **절대** 실행하지 않습니다.
- `npm install` (의존성 추가)
- `supabase db reset`
- `supabase db push`
- `git push`, `git reset --hard`, `git clean`
- `rm`, `del`, `Remove-Item`, `rmdir /s`
- 프로세스 강제 종료: `kill -9`, `Stop-Process -Force`

> **안전망**: 위 목록에 없더라도, **되돌릴 수 없거나 외부 시스템(DB·원격 저장소·배포 서버·메신저 등)에 영향을 주는 모든 명령**은 위와 동일하게 사전 확인을 거칩니다.

대신 명령을 텍스트로 보여주고 "이 명령을 실행해도 될까요?"라고 묻습니다.

**왜 이 규칙이 필요한가?**
Antigravity는 강력한 자동 실행 기능을 가집니다. 자동화는 편리하지만 실습 환경에서는 학생이 명령 한 줄 한 줄을 직접 보고 이해해야 학습이 됩니다.

## 10. 본 규칙 자기 점검 체크리스트

응답을 보내기 전에 에이전트는 다음을 스스로 확인합니다.

- [ ] 응답이 한국어인가? (규칙 1)
- [ ] 위험한 작업을 사전 동의 없이 한 게 있는가? (규칙 2)
- [ ] 복잡한 변경 전에 Plan을 보여줬는가? (규칙 3)
- [ ] 코드에 한글 주석과 설명을 충분히 달았는가? (규칙 4)
- [ ] 코드를 생략 없이 전체 제시했는가? (규칙 5)
- [ ] 한 응답에서 너무 많은 파일을 건드리지 않았는가? (규칙 6)
- [ ] SQL 파일명·위치가 규칙대로인가? (규칙 7)
- [ ] 민감 데이터를 평문으로 저장하려 하지 않았는가? (규칙 8)
- [ ] 파일을 만들기 전에 Before You Edit 체크리스트를 거쳤는가? (규칙 9)

하나라도 어긴 항목이 있으면 응답을 수정한 뒤 보냅니다.

### 10.1 자기 점검 결과 가시화 (필수)
체크 결과를 **응답의 맨 마지막 줄에 다음 형식으로 노출**합니다.

```
[Self-check] R1✓ R2✓ R3✓ R4✓ R5✓ R6✓ R7✓ R8✓ R9✓
```

- 통과한 규칙은 `✓`, 위반한 규칙은 `✗` 로 표기합니다.
- 본 응답이 단순한 정보 제공(코드 변경 없음)이라 일부 규칙이 해당 없음(N/A)이면 `-` 로 표기합니다.
  - 예: `[Self-check] R1✓ R2- R3- R4- R5- R6- R7- R8- R9-` (정보 제공만 한 경우)
- 위반(`✗`)이 하나라도 있으면 응답을 보내지 말고 **수정한 뒤 재검토**합니다.

**왜 이 규칙이 필요한가?**
"마음속으로 확인"은 LLM이 실제로 수행했는지 사용자가 검증할 수 없습니다. 한 줄 노출로 강제하면 에이전트가 매 응답마다 체크를 거치게 됩니다.
